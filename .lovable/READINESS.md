## 🔐 Security model

### Authentication
- Passwords: PBKDF2-SHA256, 600 000 iterations, 16-byte salt, 32-byte derived key.
- Hash format: `pbkdf2$<iter>$<saltB64>$<hashB64>` — fully self-describing.
- Login uses constant-time comparison + dummy hash on user-not-found to mask
  enumeration.
- JWT (jose, HS256), 7-day TTL, claims: `{ sub, tid, email, iat, exp, iss, aud }`.
- Session cookie: `HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=7d`.

### Multi-tenant isolation (dual layer)
1. **Application layer:** Every query takes a `tenantId` parameter and includes
   `WHERE tenant_id = $tid` explicitly.
2. **Database layer:** All business tables have `ENABLE ROW LEVEL SECURITY`.
   Each query runs inside a Neon HTTP transaction that first executes
   `SELECT set_config('app.tenant_id', $tid, true)`. Policies enforce
   `tenant_id = current_tenant_id()` for both `USING` and `WITH CHECK`.
3. **Bypass:** A second flag `app.tenant_bypass = 'on'` is used only by signup
   (no tenant yet) and the migration runner. Never reachable from a route.

### URL ingress (SSRF defence)
- `normalizeProjectUrl` blocks: `localhost`, `127.0.0.0/8`, `169.254.169.254`
  (cloud metadata), `10/8`, `172.16/12`, `192.168/16`, IPv6 ULA/link-local,
  `.local`, `.internal`, `.localhost`. Forces https for canonical comparison.
- `safeFetch` enforces 10 s timeout and 2 MB body cap by default.

### Ownership verification (3-tier, fail-open to next)
1. `GET https://<host>/.well-known/hn-verify-<token>.txt` — body must contain
   the token.
2. DNS TXT on `_hn-verify.<host>` via Cloudflare DoH.
3. `<meta name="hn-verify" content="<token>">` on the homepage.

Token format: 24 random bytes (hex) generated by Postgres.

### File storage
- Keys are forced into the prefix `t/<tenantId>/p/<projectId>/<name>`.
- `buildKey` rejects `..`, leading slashes, and disallowed characters.
- Download route revalidates ownership: `assertKeyForTenant(key, tenantId)`
  before streaming bytes.
- Default driver stores bytes in Postgres `bytea` (25 MB cap). Swap to R2
  later by replacing the `StorageDriver` implementation; the interface and
  key contract stay identical.

### Jobs queue
- Postgres-backed (`jobs` table), `claimBatch` uses
  `SELECT … FOR UPDATE SKIP LOCKED` for safe concurrent claim.
- Per tick: up to **10 jobs in parallel** (`runWorkerOnce(handler, 10)`).
- Retries: exponential backoff (30 s → 2 m → 8 m → 30 m → 2 h, capped),
  `max_attempts=5`, then **dead-letter** status (terminal).
- Worker can be triggered by `POST /api/jobs/run` (auth required) or by an
  external cron pinging the same endpoint.

### Input validation
- All public routes parse bodies with **Zod** (max-length, regex, ranges).
- File upload caps at 25 MB and validates filename character class.

   each only sees their own.

## 🟡 Deferred (requires Cloudflare account changes — left for last)

The functional pipeline works end-to-end with the current driver. To switch
to the originally-specified Cloudflare-native infra, follow this checklist
(no code changes inside route handlers needed — only the storage driver and a
queue consumer):

1. **R2 bucket** — create `hn-files` in your CF dashboard.
   In `wrangler.jsonc` add:
   ```jsonc
   "r2_buckets": [{ "binding": "HN_FILES", "bucket_name": "hn-files" }]
   ```
   Implement an `R2Storage` driver that reads `env.HN_FILES` and `put`/`get`
   directly via the binding. Swap `getStorage()` to return it.

2. **Cloudflare Queues** — create `hn-jobs` and `hn-jobs-dlq`. In
   `wrangler.jsonc`:
   ```jsonc
   "queues": {
     "producers": [{ "binding": "HN_JOBS", "queue": "hn-jobs" }],
     "consumers": [{
       "queue": "hn-jobs",
       "max_batch_size": 10,
       "max_concurrency": 10,
       "max_retries": 5,
       "dead_letter_queue": "hn-jobs-dlq"
     }]
   }
   ```
   Add a `queue()` handler that calls `dispatch(job)` per message. Have
   `enqueue()` ALSO send to `env.HN_JOBS.send(...)` — the DB row stays as the
   source of truth for status & UI, but Cloudflare drives execution.

3. **Cron** — add a `triggers.crons` entry hitting `/api/jobs/run` every
   minute as a belt-and-suspenders sweep for missed messages.

Until those bindings exist, the in-DB driver fulfils the same semantics
(same `StorageDriver` interface, same retry/backoff/dead-letter behaviour).
